# DineCue I18N

## Supported Languages

- English: `en`
- Turkish: `tr`
- German: `de`

## Default Fallback

If the browser/computer language is unsupported, default to `en`.

## Browser Detection

The client i18n provider normalizes `navigator.language` to `en`, `tr`, or `de`.

## Language Switcher

The public landing page and app surfaces expose a language switcher for `en`, `tr`, and `de`. Manual changes persist locally immediately and should update visible UI copy without changing routes.

## Local Persistence Before Login

Manual language changes before login are stored in local storage under `dinecue.language`.

## Account Language Precedence

After login, account `preferredLanguage` wins and updates the local language state.

Language resolution order after login/session restore is:

1. Authenticated user's `preferredLanguage`.
2. Previously selected local language.
3. Browser/computer language.
4. English fallback.

For first account creation, the language selected before login is sent through the email OTP flow and should become the account `preferredLanguage`. Manual language changes while authenticated are persisted back to the profile.

## No Locale-Prefixed Routes

Do not create `/en`, `/tr`, or `/de`. Language is a user preference, not a route structure decision in this version.

Public metadata is ready for localization, but localized SEO URLs and hreflang alternates should not be added until real localized public URLs exist.

## Recommendation Request Language

Recommendation requests send the active app language as backend `language` unless the user explicitly changes it in the request form.

## Context Controls

The find flow includes:

- `partySize`
- `withKids`
- `mealMoment`

These values are sent in the recommendation request `context` object.

## Stable Profile Values

Localized display labels must not be sent as profile values. For example, the UI may display `TĂ¼rkiye`, but profile update payloads must send the stable country code `TR`.
